# PocketPilot Codex Mobile Integration Guide

This guide defines how a mobile client integrates with the PocketPilot `codex` provider. PocketPilot controls the officially installed and authenticated Codex App Server on the computer. The mobile client connects only to PocketPilot's `/v1` API; it must not connect directly to App Server or translate Codex messages into Claude SDK messages.

This document describes the contract implemented by the current backend. PocketPilot's OpenAPI documentation is available from the local administration listener at `/documentation/`, with the raw document at `/documentation/json`. The running OpenAPI and backend version are authoritative for REST fields. The schema generated by the installed Codex App Server version is authoritative for native JSON-RPC fields.

The [Simplified Chinese Codex guide](./codex-mobile-integration-guide.zh-CN.md) covers the same integration flow.

## 1. Non-negotiable boundaries

### 1.1 The two WebSockets have different responsibilities

| Endpoint | Responsibility | Codex client behavior |
| --- | --- | --- |
| `/v1/events` | PocketPilot control events and task control state | Parse only PocketPilot control events; do not look for Codex text or items here |
| `/v1/tasks/{taskId}/agent` | Bidirectional provider-native stream for one task | Use native Codex App Server JSON-RPC frames |

The Agent WebSocket does not add an `event`, `payload`, or `kind: "sdk"` envelope. Every server text frame is one native Codex JSON object. Every client frame must be either an allowed native Codex request or a response to a native server request.

PocketPilot starts and initializes App Server over stdio internally. A mobile client must not send `initialize`, `initialized`, `thread/start`, or `thread/resume` through the task WebSocket. Conversation creation and attachment use PocketPilot REST routes. The task WebSocket accepts only the native methods listed in this guide.

### 1.2 Keep every identity separate

| Identifier | Owner | Purpose |
| --- | --- | --- |
| `taskId` | PocketPilot | REST, Agent WebSocket, task state, reconnect, and routing |
| `threadId` / `nativeConversationId` | Codex | One persistent conversation branch; the REST `conversationId` path parameter is this value |
| `sessionId` / `nativeSessionId` | Codex | Root identity of a Codex session tree; forks may share it |
| `turnId` / `activeTurnId` | Codex/PocketPilot | One currently executing user request |
| `itemId` | Codex | A user message, agent message, reasoning, command, file change, or tool item |
| JSON-RPC `id` | Requesting peer | Correlates one request and response, including approval server requests |
| `afterCursor` | PocketPilot transport | Replay position for the Agent stream; it is not a Codex thread, turn, or item ID |

`taskId` is neither `threadId` nor `sessionId`. A Codex thread can be attached to PocketPilot tasks at different times, and one task carries multiple consecutive turns.

## 2. Connection prerequisites

### 2.1 Authentication

Except for the public pairing bootstrap routes, every `/v1` request carries the access credential:

```http
Authorization: Bearer <accessToken>
```

Use the same `Authorization` header during the WebSocket handshake. Never put the credential in a URL, query string, log, analytics event, crash report, or screenshot. When converting the QR `baseUrl` to a WebSocket URL, map `http` to `ws` and `https` to `wss`.

Reuse the pairing, credential refresh, and device revocation flows from the [general mobile integration guide](./mobile-integration-guide.en.md). The Codex provider does not change PocketPilot device authentication.

### 2.2 Discover the provider and its capabilities

Call these routes before presenting Codex as available:

```http
GET /v1/providers
GET /v1/providers/codex/capabilities
```

Show Codex only when `status` is `available`. A capability snapshot has this shape; always use the protocol version returned by the server:

```json
{

  "status": "available",
  "protocolVersion": "codex-app-server@0.144",
  "capabilities": {
    "activeTurnSteering": true,
    "approvals": true,
    "attachments": false,
    "effort": true,
    "historyPagination": "cursor",
    "interrupt": true,
    "modes": true,
    "models": true,
    "nativeActions": {
      "compact": {
        "availability": "idle",
        "method": "thread/compact/start",
        "startsTurn": true
      },
      "rename": {
        "availability": "always",
        "method": "thread/name/set",
        "startsTurn": false
      },
      "review": {
        "availability": "idle",
        "deliveries": ["inline"],
        "method": "review/start",
        "startsTurn": true,
        "targetTypes": [
          "uncommittedChanges",

      }

```

Do not infer availability from a local executable path, the outcome of a previous launch, or a client-owned provider list. If Codex is unavailable, retain the returned `reasonCode` and direct the user to resolve installation or authentication on the computer.

### 2.3 Select an authorized workspace

```http
GET /v1/workspaces
```

Example response:

```json
{
  "workspaceRoots": [
    "D:\\Projects\\demo",
    "D:\\Projects\\another-repo"
  ]
}
```

Conversation listing, creation, attachment, and Agent requests all require an authorized workspace. Let the user select an exact string returned by this endpoint. Do not concatenate paths, normalize them independently, or replace path separators on the client.

## 3. Conversation list, history, and attachment

### 3.1 List Codex threads in a workspace

```http
GET /v1/providers/codex/conversations?workspace=<urlEncodedWorkspace>&limit=50&includeArchived=true&searchTerm=review
```

The optional query parameters are `cursor`, `limit`, `includeArchived=true`, and `searchTerm`. Treat `cursor` as opaque and return it unchanged. When `includeArchived=true`, PocketPilot forwards `archived: true` to App Server. When `searchTerm` is present, PocketPilot forwards it unchanged. PocketPilot asks App Server for CLI, VS Code, and App Server thread sources, then returns only threads authorized for the selected workspace.

The REST response field is `conversations`, not `items`:

```json
{
  "conversations": [
    {
      "id": "thr_019abc",
      "sessionId": "sess_019abc",
      "cwd": "D:\\Projects\\demo",
      "turns": []
    }
  ],
  "page": {
    "cursor": null,
    "hasMore": false
  }
}
```

Each entry is a native Codex thread and can gain fields in later supported versions. Preserve unknown fields, and do not reject a row merely because it contains a new property. Prefer a native Codex name when one is present. Otherwise derive a display-only fallback from the first user message or a shortened thread ID.

### 3.2 Read conversation history

```http
GET /v1/providers/codex/conversations/{threadId}?workspace=<urlEncodedWorkspace>&limit=50
```

The REST response field is `messages`:

```json
{
  "messages": [
    {
      "id": "turn_019abc",
      "status": "completed",
      "items": [
        {
          "id": "item_019abc",
          "type": "agentMessage"
        }
      ]
    }
  ],
  "page": {
    "cursor": "next-page-token",
    "hasMore": true
  }
}
```

Although the outer field is named `messages`, its entries are native Codex turns. The item variants inside each turn are owned by the installed App Server version. PocketPilot returns each page in chronological display order. Load older pages with `page.cursor` and prepend them; never reconstruct historical items as a new `turn/start` input.

Long histories require virtualization or segmented rendering. Load the latest page first. When the user reaches the top, request an older page and deduplicate by item or turn ID. The same item may arrive through history and the live stream, so reconcile it with the final `item/completed` event or the newest history result.

### 3.3 Attach an existing conversation

When the user selects a thread, call:

```http
POST /v1/providers/codex/conversations/thr_019abc/attach
Content-Type: application/json

{
  "operationId": "0c4cde63-e7a6-4a2d-a2d5-2c3a6bb16f19",
  "workspace": "D:\\Projects\\demo",
  "workspaceRiskAccepted": true
}
```

`operationId` is a UUID generated by the mobile client for this HTTP mutation. Reuse it only when retrying the same operation after an uncertain response. Generate a new value for every distinct operation.

The response is a task operation result:

```json
{
  "action": "attached",
  "task": {
    "id": "8d0d6d3b-4dc4-4c65-95e3-0a6d47a6f812",
    "provider": "codex",
    "state": "idle",
    "initialCwd": "D:\\Projects\\demo",
    "nativeConversationId": "thr_019abc",
    "nativeSessionId": "sess_019abc",
    "nativeProtocolVersion": "codex-app-server@0.144",
    "activeTurnId": null,
    "model": null,
    "permissionMode": null,
    "sdkSessionId": null,
    "origin": "agent-conversation",
    "createdAt": 1780000000000,
    "updatedAt": 1780000000000,
    "interruptedAt": null,
    "terminalAt": null
  }
}
```

If the thread is already attached to a non-terminal PocketPilot task, the backend reuses that task. Always use the returned `task.id`; never derive a task ID from the Codex thread ID.

### 3.4 Create an empty conversation

When the product exposes New conversation, call:

```http
POST /v1/providers/codex/conversations
Content-Type: application/json

{
  "operationId": "d1b3a5f1-7537-440d-a64a-5c4f2a8de8a7",
  "workspace": "D:\\Projects\\demo",
  "workspaceRiskAccepted": true
}
```

PocketPilot calls native `thread/start` on the computer and returns a new `taskId`, `nativeConversationId`, and `nativeSessionId`. This request does not submit a user prompt. Send the first user message with `turn/start` after opening the Agent WebSocket.

### 3.5 Fork a conversation

```http
POST /v1/providers/codex/conversations/{threadId}/fork
Content-Type: application/json

{
  "operationId": "00000000-0000-4000-8000-000000000040",
  "workspace": "D:\Projects\demo",
  "workspaceRiskAccepted": true
}
```

Fork creates a new native Codex thread from the selected source, creates a new PocketPilot task bound to that fork, and returns `{ action: "forked", task }`. Continue the forked conversation on `/v1/tasks/{taskId}/agent`.

### 3.6 Archive or unarchive a conversation

```http
POST /v1/providers/codex/conversations/{threadId}/archive
POST /v1/providers/codex/conversations/{threadId}/unarchive
```

Archive requires the shared operation body plus `confirm: true`. Without
confirm, archive rejects with `CONFIRMATION_REQUIRED`. Unarchive uses the same
operation body as create/attach/fork and does not require confirm. Both return
`{ action, task: null }`. Archive is reversible and does **not** auto-close a
bound PocketPilot task; close the task separately if needed. Unarchive does not
create a local task; call attach again before reconnecting the Agent WebSocket.

### 3.7 Delete a conversation

```http
POST /v1/providers/codex/conversations/{threadId}/delete
Content-Type: application/json

{
  "confirm": true,
  "operationId": "00000000-0000-4000-8000-000000000041",
  "workspace": "D:\Projects\demo",
  "workspaceRiskAccepted": true
}
```

Delete requires `confirm: true`. Without it the route rejects the request with
`CONFIRMATION_REQUIRED`. A successful delete returns
`{ action: "deleted", task: null }` and terminals any local non-terminal task
bound to that conversation. Treat delete as irreversible from the mobile client.

### 3.8 Keep thread management capability-gated

Only render fork/archive/unarchive/delete controls when `capabilities.threadManagement` advertises the corresponding boolean as `true`. Claude currently advertises all thread-management flags as `false`. Do not invent alternative REST mutation routes or send archive/delete over the Agent WebSocket.

## 4. Connect the Agent WebSocket

### 4.1 Open the connection

```text
GET /v1/tasks/{taskId}/agent
```

Platform-neutral example:

```text
WebSocket.connect(
  "ws://192.168.31.223:43182/v1/tasks/8d0d6d3b-4dc4-4c65-95e3-0a6d47a6f812/agent",
  headers = { "Authorization": "Bearer <accessToken>" },
)
```

The exact header API depends on the client library, but authentication belongs in the WebSocket handshake. PocketPilot subscribes the native stream before activating the task, so activation-time notifications are not lost.

After the connection opens, do not send `initialize`, `initialized`, `thread/start`, or `thread/resume`. These belong to PocketPilot's internal lifecycle with the local App Server.

### 4.2 Client methods currently allowed by PocketPilot

```text
turn/start
turn/steer
turn/interrupt
review/start
thread/name/set
thread/compact/start
thread/read
thread/turns/list
thread/items/list
model/list
collaborationMode/list
permissionProfile/list
```

Use REST for conversation creation, attachment, and task close. PocketPilot does not automatically expose `account/*`, configuration writes, filesystem, process, plugin, arbitrary MCP, or newly introduced methods merely because the installed App Server supports them.

PocketPilot injects the task's bound `threadId` into most requests. If the client supplies `threadId`, it must equal the task's `nativeConversationId`; a conflict is rejected. `model/list` is the catalog request that is not bound to a thread.

`turn/start`, `review/start`, `thread/compact/start`, `turn/steer`, and native server-request responses use the shared per-task P2 ordering rules. `turn/interrupt` is P1 and invalidates older active or queued P2 work before it is forwarded. Catalog and history methods are P3 reads: they do not wait behind the turn lane or consume turn capacity, but PocketPilot rechecks task availability and current workspace authorization before and after forwarding. An idle `turn/start`, review, or compact reserves the same configured active-task capacity used by Claude tasks. `thread/name/set` is always available, does not start a turn, and does not reserve capacity.

Codex remote capabilities advertise `attachments: false`. Do not invent Codex attachment input for remote tasks. Read `capabilities.nativeActions` for exact review, rename, and compact methods. Detached review is rejected; only inline review delivery is allowed.

Read `capabilities.statusCatalogs` for the closed readonly status-catalog surface. Codex currently advertises `account`, `rateLimits`, `skills`, `hooks`, and `mcpServers`. Those catalogs stay on the native Agent WebSocket as P3 reads. PocketPilot authorizes optional `cwd`/`cwds`, injects the task workspace into `skills/list` and `hooks/list` when neither path is provided, rejects `account/read` with `refreshToken: true`, and redacts email/token/path/command fields before delivery. Do not invent REST mutation routes, `account/login`, `account/logout`, plugin management, MCP install, or arbitrary path-bearing catalog writes.

### 4.3 Frame format

Codex App Server uses JSON-RPC-style objects with the `jsonrpc: "2.0"` member omitted on the wire. PocketPilot adds no envelope:

```json
{
  "id": "mobile-42",
  "method": "turn/start",
  "params": {
    "input": [
      {
        "type": "text",
        "text": "Explain the failing test.",
        "text_elements": []
      }
    ]
  }
}
```

A response echoes the request `id`:

```json
{
  "id": "mobile-42",
  "result": {
    "turn": {
      "id": "turn_019abc"
    }
  }
}
```

A notification has no `id`:

```json
{
  "method": "turn/started",
  "params": {
    "threadId": "thr_019abc",
    "turn": {
      "id": "turn_019abc"
    }
  }
}
```

A JSON-RPC error is also a native object. Preserve `code`, `message`, and unknown fields:

```json
{
  "id": "mobile-42",
  "error": {
    "code": -32600,
    "message": "Invalid request"
  }
}
```

Client request IDs need only be unique on the active connection. A counter or prefixed UUID is appropriate. Never use `taskId` as the JSON-RPC request ID.

## 5. Turn lifecycle and message submission

### 5.1 Use `turn/start` while idle

When `activeTurnId` is `null` and the task is `idle`, send:

```json
{
  "id": "mobile-43",
  "method": "turn/start",
  "params": {
    "input": [
      {
        "type": "text",
        "text": "Check this repository for TypeScript type errors.",
        "text_elements": []
      }
    ]
  }
}
```

`turn/start` is asynchronous. The response's `turn.id` correlates the request, but the later `turn/started` notification is the authoritative source for the active turn. Store `params.turn.id` from that notification as `activeTurnId`.

A typical sequence is:

```text
turn/start response
turn/started
item/started
item/agentMessage/delta        (zero or more)
item/completed
turn/completed
```

Execution may also produce commands, file changes, reasoning, plans, MCP calls, images, context compaction, and other native items. Do not assume a fixed event count or depend on optional notifications arriving in a fixed order.

PocketPilot projects these lifecycle transitions independently on `/v1/events`: `turn/started` produces `task.state = executing`, a pending native server request produces `awaiting_approval`, resolving the last request returns to `executing`, and `turn/completed` returns to `idle`. Use these control events for shared task chrome, while the native Agent frames remain the conversation source of truth.

### 5.2 Use `turn/steer` during an active turn

When a turn is active, additional user input must not start another turn. Send:

```json
{
  "id": "mobile-44",
  "method": "turn/steer",
  "params": {
    "expectedTurnId": "turn_019abc",
    "input": [
      {
        "type": "text",
        "text": "Focus on parser-related files first.",
        "text_elements": []
      }
    ]
  }
}
```

`expectedTurnId` must equal the ID from the latest `turn/started` notification. PocketPilot validates it; a missing or stale value produces a task-busy failure. Codex has no Claude `priority` or `shouldQuery` fields, so do not send or translate them.

### 5.3 Interrupt and close

The normal Stop action should call:

```http
POST /v1/tasks/{taskId}/interrupt
Content-Type: application/json

{
  "operationId": "f97fa0c3-b1b4-48b5-8e12-ec7e88a9766f"
}
```

PocketPilot invokes native `turn/interrupt(threadId, turnId)` with the current `activeTurnId`. The Agent WebSocket also allows the native request:

```json
{
  "id": "mobile-45",
  "method": "turn/interrupt",
  "params": {
    "turnId": "turn_019abc"
  }
}
```

Both forms require the current active turn. Native `turn/completed` remains authoritative after an interrupt; do not mark the UI complete merely because the REST operation returned successfully.

Close a task with `POST /v1/tasks/{taskId}/close`. Closing makes the PocketPilot task terminal, but it does not archive or delete the Codex thread. To continue later, list and attach that thread again.

## 6. Consume streaming items

### 6.1 Text deltas

Codex normally streams agent text with `item/agentMessage/delta`:

```json
{
  "method": "item/agentMessage/delta",
  "params": {
    "threadId": "thr_019abc",
    "turnId": "turn_019abc",
    "itemId": "item_019abc",
    "delta": "Inspecting the test files"
  }
}
```

Keep a delta buffer keyed by `itemId` and append `delta` immediately. When `item/completed` arrives for that item, reconcile the buffer with the complete native item. If no delta arrived, do not fabricate token-by-token animation.

### 6.2 Other native items

The current reviewed notification set includes:

```text
item/started
item/completed
item/commandExecution/outputDelta
item/commandExecution/terminalInteraction
item/fileChange/patchUpdated
item/mcpToolCall/progress
item/plan/delta
item/reasoning/summaryPartAdded
item/reasoning/summaryTextDelta
item/reasoning/textDelta
turn/diff/updated
turn/plan/updated
thread/status/changed
thread/tokenUsage/updated
```

These are native Codex method names. Use an extensible dispatch keyed by `method`: render known variants and preserve or safely ignore unknown variants. Do not map them to Claude `assistant`, `tool_use`, or `stream_event` messages.

`item/completed` is the authoritative final item and `turn/completed` is the authoritative final turn. A client can render deltas optimistically and replace them with the complete native object. Use `(threadId, turnId, itemId)` as the live-item deduplication key.

## 7. Models, effort, modes, and permissions

Do not reuse Claude model, effort, or permission-mode enums. Query the catalogs supplied by the installed Codex version after opening the Agent WebSocket:

```json
{
  "id": "catalog-models",
  "method": "model/list",
  "params": {}
}
```

```json
{
  "id": "catalog-modes",
  "method": "collaborationMode/list",
  "params": {}
}
```

```json
{
  "id": "catalog-permissions",
  "method": "permissionProfile/list",
  "params": {}
}
```

Responses retain their Codex-native structure and normally expose the page under `result.data`. Do not hard-code item fields or enum values. Render only the models, supported reasoning efforts, collaboration modes, and permission profiles returned by the current server.

Pass the user's selection as native parameters on the next `turn/start` or supported turn request:

```json
{
  "id": "mobile-46",
  "method": "turn/start",
  "params": {
    "model": "<value from model/list>",
    "effort": "<supportedReasoningEfforts value for the selected model>",
    "collaborationMode": "<native value from collaborationMode/list>",
    "approvalPolicy": "<native Codex policy>",
    "permissions": "<profile id from permissionProfile/list>",
    "sandboxPolicy": "<native value authorized for this workspace>",
    "input": [
      {
        "type": "text",
        "text": "Start working on this issue.",
        "text_elements": []
      }
    ]
  }
}
```

The angle-bracket strings are explanatory placeholders and must never be sent literally. The exact types of `model`, `effort`, `collaborationMode`, `permissions`, and `sandboxPolicy` come from the current experimental App Server schema. Changing these choices does not create a new PocketPilot task or change the Codex thread ID.

Do not call the Claude-only `/composer-options`, `/model`, `/effort`, `/permission-mode`, or approval REST controls for a Codex task. They return `TASK_CONTROL_NOT_SUPPORTED`; Codex selections and approval responses stay native on the Agent WebSocket.

PocketPilot does not advertise a fixed Codex slash-command panel. Do not copy Claude `/clear`, `/compact`, `/review`, `/security-review`, `/code-review`, or aliases into Codex. Creating a new Codex conversation uses the provider conversation REST operation and native `thread/start`; selecting an existing conversation continues its native thread.

Use the Codex-native methods advertised by `capabilities.nativeActions`:

```json
{
  "id": "mobile-60",
  "method": "review/start",
  "params": {
    "delivery": "inline",
    "target": { "type": "uncommittedChanges" }
  }
}
```

```json
{
  "id": "mobile-61",
  "method": "thread/name/set",
  "params": {
    "name": "Provider parity audit"
  }
}
```

```json
{
  "id": "mobile-62",
  "method": "thread/compact/start",
  "params": {}
}
```

Review and compact require an idle task, share turn capacity with `turn/start`, and move the task to `executing` when `turn/started` arrives. Detached review is rejected. Rename is always available and does not start a turn.

## 8. Native Codex approvals and user input

Codex approvals are not Claude `PermissionResult` objects. The unchanged native server request arrives on the Agent WebSocket:

```json
{
  "id": "approval-42",
  "method": "item/commandExecution/requestApproval",
  "params": {
    "threadId": "thr_019abc",
    "turnId": "turn_019abc",
    "itemId": "item_019abc",
    "startedAtMs": 1780000000000,
    "command": "git status"
  }
}
```

PocketPilot also publishes a rendering projection on `/v1/events`:

```json
{
  "type": "event",
  "event": {
    "cursor": 42,
    "taskId": "8d0d6d3b-4dc4-4c65-95e3-0a6d47a6f812",
    "occurredAt": 1780000000000,
    "event": {
      "kind": "approval.requested",
      "payload": {
        "provider": "codex",
        "requestId": "approval-42",
        "method": "item/commandExecution/requestApproval",
        "params": {
          "threadId": "thr_019abc",
          "turnId": "turn_019abc",
          "itemId": "item_019abc",
          "command": "git status"
        }
      }
    }
  }
}
```

This control event is only a UI projection. It does not replace or wrap the native request. Match it by `requestId`, then send the method-specific result on the Agent WebSocket using the same native JSON-RPC `id`.

PocketPilot currently forwards these server request methods:

```text
item/commandExecution/requestApproval
item/fileChange/requestApproval
item/permissions/requestApproval
item/tool/requestUserInput
mcpServer/elicitation/request
```

Display the method's native fields and return its method-specific native result using the same JSON-RPC `id`. For example, the current command-approval schema accepts:

```json
{
  "id": "approval-42",
  "result": {
    "decision": "accept"
  }
}
```

Result schemas differ by method. Generate production response types from the current Codex App Server schema and preserve unrecognized fields. Do not collapse all approvals into `{ allowed: true }`, Claude `behavior: "allow"`, or a custom `approvalId` contract.

Several server requests can be pending concurrently. Keep a pending map keyed by JSON-RPC `id`; neither `turnId` nor `itemId` replaces that key. A request becomes stale after turn completion, interrupt, task close, workspace revocation, or an App Server restart. On `STALE_APPROVAL`, remove the local pending request and do not retry its response.

## 9. Workspace and path authorization

PocketPilot validates path-bearing Codex request fields against the selected workspace. For `turn/start` and `turn/steer`, this includes `cwd`, `runtimeWorkspaceRoots`, sandbox roots, paths inside environment objects, and `localImage`, `skill`, or `mention` input paths. Named paths in approval requests are also validated.

Ordinary prompt text is not interpreted as a path, so the user can mention text such as `D:\\outside\\file.ts`. Do not alter prompt text in an attempt to perform client-side authorization. Send only the selected authorized workspace as `workspace`, and continue using the native Codex sandbox and permission profile. PocketPilot workspace authorization is not a replacement for the Codex sandbox.

## 10. Disconnect, reconnect, and replay

### 10.1 Reconnect sequence

1. When the Agent WebSocket disconnects, call `GET /v1/tasks/{taskId}` before deciding what to send next.
2. Stop sending if the task is `terminal`. If it is `interrupted`, follow the product's explicit resume or reattach flow.
3. Discard the local in-progress Codex turn projection, including delta buffers, before reopening `/v1/tasks/{taskId}/agent` without `afterCursor`.
4. Rebuild the in-progress UI from the complete retained active-turn replay, then continue with live notifications. Do not append that replay to the pre-disconnect delta buffers.
5. If no active turn is retained or a final `turn/completed` was missed, read native history and replace the final item list from that source of truth.
6. Never automatically resend a `turn/start` after an uncertain response. First inspect `turn/started`, `turn/completed`, task metadata, and native history to determine whether Codex already executed it.

### 10.2 Current `afterCursor` limitation

`afterCursor` is PocketPilot transport metadata, not a field in a Codex frame:

```text
GET /v1/tasks/{taskId}/agent?afterCursor=<opaque-cursor>
```

A known retained cursor replays later frames for compatibility. A missing, malformed, expired, or evicted cursor replays the complete retained active turn from its beginning. Starting a newer turn resets that window; completing, interrupting, closing, or revoking the task clears it. The current PocketPilot Agent stream does not add cursor metadata to native JSON frames and does not return the latest cursor during the WebSocket handshake.

For the current client, omit `afterCursor`. On reconnect, clear the old in-progress reducer state and apply the full active-turn replay once. Native IDs still identify items and terminal notifications, but they are not permission to append text or reasoning deltas to the old reducer a second time. Reconcile completed work from history. Never substitute `threadId`, `turnId`, `itemId`, or a message UUID for the transport cursor.

### 10.3 App Server restart

An App Server process restart creates a new bridge generation. PocketPilot initializes the new internal connection and calls native `thread/resume` before forwarding new work for an attached task. Approval request IDs from the old generation are not transferred; all old pending requests must become stale.

## 11. Errors and close codes

### 11.1 Common REST errors

The mobile client should recognize at least these stable error codes:

| Error code | Meaning | Recommended handling |
| --- | --- | --- |
| `AGENT_PROVIDER_UNAVAILABLE` | Codex is not installed, ready, or available | Refresh provider state and direct the user to the computer |
| `CODEX_THREAD_NOT_FOUND` | The thread is missing, unreadable, or outside the workspace | Remove the stale row and reload the conversation list |
| `CODEX_HISTORY_UNAVAILABLE` | Native history is temporarily unavailable | Keep the current UI and retry history later; do not send history as a prompt |
| `WORKSPACE_NOT_AUTHORIZED` | The workspace or a path is outside authorized roots | Stop the request and ask the user to select an authorized workspace |
| `TASK_BUSY` | The turn precondition is stale or the current state rejects the action | Refresh task and active-turn state before deciding what to do |
| `CONCURRENT_TASK_LIMIT_REACHED` | Starting this idle turn would exceed shared Claude/Codex capacity | Keep the composer input and retry after another active task becomes idle |
| `TASK_OPERATION_SUPERSEDED` | A P0 close/revocation or P1 interrupt invalidated an older queued operation | Drop that old operation; never retry it against the newer task generation automatically |
| `TASK_CONTROL_NOT_SUPPORTED` | A Claude-only REST control was used for a Codex task | Use Codex native catalogs, turn parameters, or native approval responses |
| `STALE_APPROVAL` | The server request is no longer pending | Remove the local approval and do not replay it |
| `CODEX_APP_SERVER_UNAVAILABLE` | App Server is unavailable or returned an invalid contract | Show provider unavailability and wait for a reconnect |

### 11.2 Agent WebSocket close codes

| WebSocket code | Reason | Handling |
| ---: | --- | --- |
| `4000` | `SDK_MESSAGE_INVALID` | Check for invalid JSON, a binary frame, or a non-allowlisted method |
| `4003` | `AUTHENTICATION_FAILED` | Refresh credentials or pair again |
| `4004` | `TASK_NOT_FOUND` | Fetch tasks again or reattach the native thread |
| `4009` | `TASK_SESSION_UNAVAILABLE` | Refresh task state; interrupted and terminal tasks cannot open this stream |
| `4011` | `SDK_TRANSPORT_FAILED` | The provider bridge failed; reconnect with exponential backoff and jitter |

The `SDK` wording in these reason strings is retained for transport compatibility. For a Codex task it describes the provider-native transport failure; it does not mean the frame is a Claude SDK message.

## 12. Recommended client state machine

A mobile client can use this minimal state model without creating another task abstraction:

```text
select provider
  -> select workspace
  -> GET conversations
  -> select thread / create conversation
  -> attach or create; save taskId + threadId + sessionId
  -> GET latest history page
  -> open Agent WebSocket
  -> idle: turn/start
  -> executing: consume native notifications and items
  -> active input: turn/steer(expectedTurnId)
  -> approval server request: return native result with the same id
  -> turn/completed: clear activeTurnId and return to idle
  -> disconnect: refresh task, discard the active projection, and rebuild from full active-turn replay
```

Use `taskId` as the transport and runtime-cache key. Use `threadId` as the Codex conversation key. The UI can present one conversation, but the client must not merge those identifiers. Model, effort, mode, and permission choices affect later native turns without creating a new thread or task.

## 13. Differences from the Claude provider

| Area | Codex | Claude |
| --- | --- | --- |
| Agent WebSocket frame | Codex App Server JSON-RPC-style object | Raw Claude Agent SDK `SDKUserMessage` / `SDKMessage` |
| Initialization | PocketPilot handles it internally; mobile sends none | PocketPilot creates or resumes the SDK Query internally |
| History | Native turns from `thread/turns/list`, wrapped by REST as `messages` | SDK `SessionMessage` history |
| Streaming text | `item/agentMessage/delta` plus `item/completed` | SDK `stream_event`, `content_block_delta`, and related messages |
| Active input | `turn/steer(expectedTurnId)` | Native Claude SDK message behavior |
| Approvals | Native server requests resolved by JSON-RPC ID; `/v1/events` adds a provider-tagged rendering projection | Claude SDK `PermissionResult` and PocketPilot approval controls |
| Model and mode catalogs | `model/list`, `collaborationMode/list`, `permissionProfile/list` | Claude SDK/task composer options |
| Identity | `taskId` is separate from Codex `threadId` | `taskId` is separate from Claude `sdkSessionId` |

Read `task.provider` and `nativeProtocolVersion` before selecting a codec. Never infer the provider from a path, history shape, or a message's `type` member.

## 14. Complete request sequence

The shortest successful flow for an existing thread is:

```text
1. GET /v1/providers
2. GET /v1/providers/codex/capabilities
3. GET /v1/workspaces
4. GET /v1/providers/codex/conversations?workspace=...
5. GET /v1/providers/codex/conversations/{threadId}?workspace=...&limit=50
6. POST /v1/providers/codex/conversations/{threadId}/attach
7. Save task.id, task.nativeConversationId, and task.nativeSessionId
8. GET /v1/tasks/{taskId}
9. Open /v1/tasks/{taskId}/agent
10. Send turn/start
11. Reduce deltas and item/completed by itemId
12. Store activeTurnId from turn/started
13. Use turn/steer while activeTurnId exists
14. Clear activeTurnId and update history after turn/completed
```

For a new thread, replace steps 5 and 6 with `POST /v1/providers/codex/conversations`. Do not send a placeholder prompt merely to create an empty conversation UI.
